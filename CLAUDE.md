# 编码协作全规则
请严格遵守以下要求，执行规划、编码、迭代修复、验证全流程工作。

## 1. 结构原则
优先使用**最简实现方案**，能用基础逻辑完成需求，不做多层嵌套、复杂封装与过度设计。

## 2. 迭代修改规则
- 已验证、运行正常的代码与逻辑，**禁止无故删除、替换、改写**
- 仅针对问题做局部调整，大范围改动前必须先说明原因
- 原有稳定模块统一保留，不随意变更

## 3. 方案规划要求（含测试设计）
1. 规划阶段全面梳理细节：正常流程、分支场景、边界条件、异常处理、数据流转与模块调用关系，保障方案可落地。
2. 方案末尾**必须配套完整、严谨的测试内容**。
3. 测试禁止只设计简易正向用例、仅为强行通关；需多角度设计用例，全面校验逻辑合理性、稳定性与可靠性。
4. 测试范围覆盖常规场景、临界边界、非法输入、异常分支、历史重复问题点，做到全维度校验。

## 4. 重复问题处理
- 修复问题**必须定位根因**，不做临时表面修复
- 若同一问题反复出现，主动补充校验、拦截、容错逻辑，从根源规避复发
- 每次修复完成后，写明：修复内容 + 防重复出现的保障措施

## 5. 代码执行过程
- 当代码能够安全顺利运行时，自动执行；仅在存在高风险操作或需要用户补充信息时，才向用户提供选择项。

## 6. 输出规范
所有改动标注位置与作用，逻辑、测试描述清晰直白。
